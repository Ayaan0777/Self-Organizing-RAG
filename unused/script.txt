"Step 1 — Text Extraction: We support three file formats — PDF (using

"Step 2 — Text Cleaning: The extracted text goes through a cleaning

[CODE REFERENCE: controllers/ingestion.py — lines 20-25 (clean_text)]

"Step 3 — Recursive Splitting: We use LangChain's
RecursiveCharacterTextSplitter with these parameters:

  - CHUNK_SIZE = 1250 characters maximum per chunk
  - CHUNK_OVERLAP = 200 characters overlap between consecutive chunks
  - That's 16% overlap, within the industry standard of 10-20%

The key innovation is the SEPARATOR HIERARCHY. The splitter doesn't just
cut at character positions. It tries separators in this order:

  1. '\\n\\n' — paragraph breaks (best semantic boundary)
  2. '\\n'   — line breaks
  3. '. '    — sentence endings
  4. '? '    — question marks
  5. '! '    — exclamation marks
  6. '; '    — semicolons (clause boundaries)
  7. ', '    — commas
  8. ' '     — individual words
  9. ''      — character boundary (last resort)

This means the splitter first tries to keep whole paragraphs together.
If a paragraph is too long, it splits at sentences. Only as a last resort
does it break mid-word. This preserves semantic coherence far better than
fixed-position splitting."

[CODE REFERENCE: controllers/ingestion.py — lines 11-17]

"Step 4 — Minimum Size Enforcement: After splitting, we enforce a
MINIMUM chunk size of 500 characters. Any chunk smaller than 500 gets
merged with its neighbour.

The algorithm works like this:
  1. Start with the first chunk as a 'buffer'
  2. For each subsequent chunk: if the buffer is under 500 chars,
     merge the next chunk into the buffer
  3. After the loop, if the trailing buffer is still under 500 chars,
     merge it backward with the last accepted chunk

This prevents tiny fragments like '(see Table 3)' from becoming their
own chunks — they'd produce poor embeddings and waste retrieval slots."

[CODE REFERENCE: controllers/ingestion.py — lines 28-66]

"Step 5 — Batched Upload: The final chunks are uploaded to Pinecone in
batches of 50 to avoid overwhelming the Ollama embedding model. Each
chunk gets embedded by mxbai-embed-large into a 1024-dimensional vector."

[SHOW: Terminal output showing:
"Chunk stats: X chunks | min=XXX, max=XXX, avg=XXX chars"]

"As you can see, every chunk is between 500 and 1250 characters. The
system logged the exact statistics — minimum size, maximum size, and
average. If any tiny chunks were merged, that's logged too."

[14:30 - 20:00]  FEATURE 4 — Dynamic K Chunks during Retrieval
================================================================================

[SHOW: Dashboard → "Ask Query" page]

SPEAK:
"Now let's talk about HOW MANY chunks we retrieve. In a traditional RAG
system, you always retrieve a fixed number — typically top 5 or top 10.
But this is wasteful and sometimes harmful:

  - A simple question like 'Who invented the telephone?' might only need
    2 highly relevant chunks. Retrieving 5 means 3 are noise.
  - A complex question like 'Compare the economic policies of the US and
    China' might need 8-10 chunks spanning multiple topics.

Our system uses Dynamic K Selection — it picks the optimal K for each
query. Let me show you the 3-stage algorithm."

[SHOW: Code or diagram of the 3-stage algorithm]

"STAGE 1 — Query Complexity Analysis:
  First, we classify the query into one of three categories. The
  classifier uses EMBEDDING-BASED SIMILARITY — not keywords.

  It works like this:
  1. We have 30 pre-written prototype questions — 10 per category:
     - short_factual: 'What year?', 'Who invented?', 'Name the capital'
     - complex: 'Explain how...', 'Describe the process...', 'Why did...'
     - cross_section: 'Compare X and Y', 'How do these differ?'

  2. All 30 prototypes are embedded once (lazily on first call, then
     cached in memory).

  3. When a query comes in, we embed it and compute the MEAN cosine
     similarity to each category's 10 prototypes.

  4. The category with the highest mean similarity wins.

  There's also a keyword-based fallback if the embedding model is
  unavailable — it checks for words like 'who', 'what', 'compare',
  'explain' and the query's word count.

  Each category maps to K bounds:
    - short_factual: K range 2-4
    - complex: K range 5-8
    - cross_section: K range 6-10"

[CODE REFERENCE: controllers/metrics.py — lines 251-392
(prototype definitions + classifier)]

[ACTION: Type "Who invented binary search?" and execute]

"For this short factual question, the classifier identifies it as
'short_factual' — K range 2-4."

"STAGE 2 — Score Threshold Pruning:
  After retrieving the maximum K chunks, we prune any chunk scoring
  below 0.25 (the SCORE_NOISE_FLOOR). These are pure noise — the
  embedding isn't even in the right neighbourhood.

  If this pruning drops us below k_min, we keep at least k_min chunks."

"STAGE 3 — Score Cliff Detection:
  We look at the scores in descending order and find NATURAL DROP-OFFS.
  If the gap between any two consecutive chunks exceeds 0.12 (the
  SCORE_CLIFF_THRESHOLD), we cut there.

  Example: scores are [0.82, 0.78, 0.75, 0.45, 0.38]
  - Gap between 0.75 and 0.45 is 0.30 — that's way above 0.12
  - So we cut after position 3, keeping only 3 chunks

  The final K is clamped to the category's [k_min, k_max] bounds."

[22:00 - 28:00]  THE 5 DETECTION RULES
================================================================================

[SHOW: Code or diagram showing the 5 rules]

SPEAK:
"RULE 1 — low_top_score (Score-based, no LLM call):
  Checks if the TOP-1 retrieval score is below 0.65.

  Why 0.65? Our embedding model is mxbai-embed-large, which produces
  similarity scores in the 0.55–0.85 range. A typical 'good match' is
  0.70+. If even the BEST chunk scores below 0.65, the retrieval
  completely failed to find relevant content.

  This is the simplest and cheapest check — just compare a number."

[CODE REFERENCE: detector/detectors.py — line 22, lines 159-161]

"RULE 2 — score_drop (Score-based, no LLM call):
  Looks at the MAXIMUM GAP between any two adjacent retrieval scores.
  If that gap exceeds 0.15, there's a retrieval cliff.

  Example: scores [0.80, 0.78, 0.42, 0.38]
  The gap between 0.78 and 0.42 is 0.36 — way above 0.15.
  This means we retrieved some relevant chunks AND some completely
  irrelevant ones. The noise chunks will confuse the LLM.

  Note: we use MAX ADJACENT GAP, not rank-1 minus rank-K. The latter
  scales with K — when dynamic K is promoted, K=2 gives a smaller
  spread than K=10, making the rule's sensitivity drift. Max adjacent
  gap is K-invariant."

[CODE REFERENCE: detector/detectors.py — line 23, lines 167-170]

"RULE 3 — llm_uncertainty (Text pattern, no LLM call):
  Scans the LLM's response for HEDGING LANGUAGE. We have a list of
  50+ uncertainty phrases including:

  Direct uncertainty: 'i don't know', 'i'm not sure', 'cannot find',
  'no information', 'unable to find'

  Refusal patterns: 'does not provide', 'does not contain', 'does not
  mention', 'not mentioned', 'not specified', 'no specific', 'no
  evidence', 'cannot determine', 'not found in'

  Document refusals: 'the text does not', 'the context does not',
  'the passage does not', 'based on the provided'

  If ANY of these phrases appear in the lowercased response, the rule
  fires. The logic is: if the chunk contains the answer, the LLM just
  states it directly. Any hedging = the answer isn't in the chunks."

[CODE REFERENCE: detector/detectors.py — lines 34-52, lines 172-174]

"RULE 4 — semantic_mismatch (Embedding-based, costs 1 Ollama call):
  This rule checks if the retrieved chunks are SEMANTICALLY COHERENT
  with each other. It works like this:

  1. Embed all retrieved chunks (batched in a single Ollama call)
  2. Compute ALL pairwise cosine similarities between chunks
  3. Calculate the mean pairwise similarity
  4. Compare against a RELATIVE threshold: 0.65 × top1_score

  Why relative instead of absolute? Cross-section queries naturally
  pull chunks from different sub-topics within one document — like
  stadium, game, and players paragraphs from a Wikipedia article. Their
  mean pairwise sim sits ~0.55–0.65, which would trip any absolute
  threshold above ~0.55.

  By tying the bar to top1_score, we expect coherence proportional to
  retrieval quality: high top1 → expect tight chunks; modest top1 →
  tolerate more spread.

  The rule skips when top1 ≤ 0 (no retrieval) — Rule 1 owns that case."

[CODE REFERENCE: detector/detectors.py — lines 24, 69-102]

"RULE 5 — evidence_mismatch (Embedding-based, costs 1 Ollama call):
  This is our hallucination detector. It checks whether the LLM's
  answer is GROUNDED in any retrieved chunk.

  1. Embed the answer and each chunk in a single batched call
  2. Compute the answer's cosine similarity to EACH chunk individually
  3. Take the MAXIMUM similarity (best-matching chunk)
  4. If the max is below 0.60, the answer isn't backed by anything

  Why per-chunk and not concatenated? A short factual answer like
  '$5,000,000' embeds far from a long concatenated context blob purely
  due to length asymmetry. The old 'embed all chunks together' version
  flagged correct answers reflexively. 'Grounded in SOME chunk' is what
  we actually mean."

[CODE REFERENCE: detector/detectors.py — lines 32, 105-135]

[32:30 - 35:30]  THE 4-STRATEGY REPAIR CASCADE
================================================================================

[SHOW: repair/cascade.py — strategy overview]

SPEAK:
"When repair is triggered, each event goes through a 4-STRATEGY ORDERED
CASCADE. The strategies are tried in order S1 → S2 → S3 → S4. The FIRST
strategy that resolves the issue wins, and the cascade stops.

If all four fail, the event is marked UNFIXABLE."

[SHOW: Strategy diagram or table]

"STRATEGY 1 — Dynamic K Selection (s1_dynamic_k):
  Varies the retrieval K based on question category WITHOUT touching
  Pinecone. This is a READ-ONLY experiment.

  It classifies the query, picks a dynamic K from the category bounds,
  and re-probes. If retrieving a different number of chunks alone
  produces a better answer — we're done.

  This is the cheapest strategy — no Pinecone writes, no rechunking."

"STRATEGY 2 — Chunk Size Variation (s2_chunk_size):
  The decision engine DIAGNOSES the root cause and recommends a new
  chunk configuration. K is FIXED at 5 for fair comparison.

  The diagnosis analyzes:
    - Which detectors fired
    - The query's question category
    - Stage 2 metrics (precision, sufficiency, hallucination rate)
    - Retrieval scores

  Based on the diagnosis, it recommends one of 5 chunk configs:

  ┌────────────────────────┬────────────┬─────────┬───────────┐
  │ Root Cause             │ Chunk Size │ Overlap │ Algorithm │
  ├────────────────────────┼────────────┼─────────┼───────────┤
  │ High hallucination     │    200     │   40    │    llm    │
  │ Low precision (factual)│    350     │   70    │ semantic  │
  │ Insufficient context   │   1400     │  250    │ semantic  │
  │ Cross-section failure  │   1700     │  300    │  entropy  │
  │ Stale content          │  (keep)    │ (keep)  │ semantic  │
  └────────────────────────┴────────────┴─────────┴───────────┘

  Three different RECHUNKING ALGORITHMS are available:
    - semantic: RecursiveCharacterTextSplitter (configurable size/overlap)
    - llm: Asks the LLM to identify topic boundary sentences
    - entropy: Splits at sentences with high vocabulary novelty (>60% new
      words = topic shift)"

[CODE REFERENCE: detector/decision_engine.py (diagnosis),
repair/chunker.py (rechunking algorithms)]

"STRATEGY 3 — Combined (s3_combined):
  Uses BOTH dynamic K AND rechunking together. Sometimes the problem
  needs both a different K and different chunk sizes.

STRATEGY 4 — Alternate LLM (s4_alt_llm):
  Same chunks, no Pinecone changes. Swaps mistral for gemma3:27b —
  a larger, more capable model.

  S4 has a DIFFERENT win condition: it doesn't use _is_improved()
  (since the retrieval scores are unchanged). Instead, S4 wins if
  the fallback LLM produces a SUBSTANTIVE answer — meaning it's not
  a refusal like 'I don't know' or 'The text does not mention this'.

  This handles cases where the chunks DO contain the answer but the
  smaller model can't extract it."

================================================================================

[SHOW: repair/reembedder.py or terminal repair logs]

SPEAK:
"Strategies S2 and S3 modify Pinecone — they delete old chunks and insert
new ones. This is DANGEROUS. If the repair makes things worse, we could
permanently damage the knowledge base.

That's why we have a SNAPSHOT + ROLLBACK system:"

"Step 1 — SNAPSHOT: Before deleting any chunks, the reembedder saves a
  complete copy to the ChunkSnapshot table in SQLite. This includes:
    - The Pinecone vector ID
    - The original chunk text
    - The full metadata as JSON
    - The namespace

  This is our insurance policy — we can always restore the original
  state."

"Step 2 — RECHUNK: The old chunks are deleted from Pinecone. The text is
  rechunked with the new strategy and parameters. The new chunks are
  embedded and inserted into Pinecone."

"Step 3 — PROBE: We re-run the ORIGINAL query and measure metrics at the
  SAME K value. This gives us an apples-to-apples comparison."

"Step 4 — JUDGE: The _is_improved() function checks:
  - Did the top-1 score improve? (by at least 0.001)
  - Did precision, recall, or accuracy improve? (by at least 0.01)
  - Did any metric DEGRADE significantly? (>0.1 drop = reject)
  - Is the new answer a REAL answer? (not a refusal/hedging)

  CRITICAL: Even if scores improved, if the answer is still 'I don't
  know' — the repair is REJECTED. A better score that still produces a
  non-answer is not a real improvement."

"Step 5 — COMMIT or ROLLBACK:
  If improved → keep the new chunks, mark the event as resolved.
  If not improved → ROLLBACK:
    1. Delete the new chunks from Pinecone
    2. Load the snapshot entries from SQLite
    3. Re-embed each original chunk with the current model
    4. Upsert them back to Pinecone with the original vector IDs
    5. Clean up the snapshot entries

  Old data is NEVER permanently lost."

[ACTION: Show the expected JSON format]

"The evaluation runs the full RAG pipeline on each question and computes
7 METRICS per question:

STAGE 1 — Traditional Metrics (4):
  1. ROUGE-L: Longest common subsequence F1 between generated answer
     and ground truth. Measures word-level overlap.
  2. Semantic Similarity: Embedding cosine similarity between answer
     and GT. Captures meaning even with different wording.
  3. Context-Question Similarity: Mean cosine sim between retrieved
     chunks and the question. Are the chunks relevant to what was asked?
  4. Context-GT Similarity: Mean cosine sim between chunks and GT.
     Do the chunks contain the actual answer?

STAGE 2 — Self-Healing Metrics (3):
  5. Retrieval Precision@K: What fraction of retrieved chunks are
     relevant? A chunk is 'relevant' if its similarity to any ground
     truth ≥ 0.50.
  6. Context Sufficiency: Boolean — can the combined context fully
     answer the question? (combined context ↔ GT similarity ≥ 0.70)
  7. Hallucination Rate: What fraction of answer claims are NOT grounded
     in the retrieved context? Each sentence is checked against each
     chunk. If the best match < 0.55, that claim is ungrounded.

All Stage 2 metrics are EMBEDDING-BASED — no extra LLM calls needed.
Results are saved to a CSV file AND persisted to the EvalSnapshot table
in the dashboard database."

[46:00 - 50:00]  CONCLUSION — The Self-Organising Loop
================================================================================

[SHOW: Dashboard Overview page with updated stats]

SPEAK:
"Let me bring everything together. Here's the complete self-organising
loop that our system implements:

 1. INGEST — Documents are chunked with variable sizes (500-1250 chars),
    using a semantic separator hierarchy and minimum-size enforcement.

 2. GATE — Each query passes through a 2-tier retrieval gate that
    filters greetings and chitchat, saving unnecessary vector searches.

 3. RETRIEVE — For real questions, the pipeline retrieves top-K chunks.
    After Strategy 1 promotion, Dynamic K selects the optimal K based
    on query complexity and score patterns.

 4. ANSWER — A single LLM call generates the answer using only the
    retrieved context, with a strict factual-only prompt.

 5. DETECT — On a background thread, 5 detection rules check for low
    retrieval score, score drops, LLM uncertainty, semantic mismatch,
    and evidence mismatch.

 6. ENRICH — If the query matches ground-truth data, precision,
    sufficiency, and hallucination metrics are computed and stored.

 7. MONITOR — The auto_worker daemon polls every 5 seconds. When ≥5
    pending events AND ≥30% of total queries are flagged, batch repair
    kicks in.

 8. REPAIR — The 4-strategy cascade tries Dynamic K → Chunk Size →
    Combined → Alt LLM. Each attempt is probed, judged, and either
    committed or rolled back with full provenance.

 9. PROMOTE — When Dynamic K accumulates 5 successes, it's promoted
    to the main pipeline — the system LEARNS from its repairs.

10. INDEX — The auto-indexer detects and refreshes stale embeddings
    via in-place upsert, maintaining index health over time.

11. EXPAND — New knowledge can be added anytime through the add-chunks
    endpoint without re-ingesting existing documents.

The result is a RAG system that doesn't just retrieve and generate —
it monitors, detects, diagnoses, heals, learns, and organises itself.

Thank you."
